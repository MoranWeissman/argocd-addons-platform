# Bug Fixes — 2026-03-30

## Bug 1: Dashboard "degraded apps" link shows nothing useful
**Problem:** Clicking "X degraded applications" on dashboard navigates to `/addons?filter=unhealthy` but the add-ons catalog doesn't show WHICH specific apps are degraded, on which clusters, or what the error is.

**What it should do:** Show a detailed breakdown — addon name, cluster, health status, sync status, and the actual error/condition from ArgoCD. Either:
- A dedicated "Issues" panel on the dashboard itself (no navigation needed)
- Or navigate to a filtered view that actually shows the degraded apps with details

**Needs discussion:** Should this be a dashboard panel or a filtered page?

---

## Bug 2: Dashboard "disconnected clusters" link doesn't filter
**Problem:** Clicking "X disconnected clusters" navigates to `/clusters?status=disconnected` but the Clusters page doesn't read the URL query param to pre-apply the filter.

**Fix:** Read `status` query param from URL in ClustersOverview and apply it as initial filter.

**Effort:** 15 min

---

## Bug 3: Failed cluster shows "healthy" addons
**Problem:** When clicking a cluster with failed connection, addons on that cluster show as "Healthy" — which is impossible since the cluster is unreachable. No info about the cluster connection issue is shown. No way to see details about individual addons.

**What it should do:**
- Show cluster connection status prominently at the top of cluster detail
- If cluster is disconnected/failed, show a warning banner: "This cluster is unreachable — addon health data may be stale"
- Addon health should reflect the ArgoCD app conditions (see bug 5)

**Needs discussion:** Should we mark all addons as "Unknown" when cluster is disconnected?

---

## Bug 4: Addons catalog shows "Not Deployed" confusingly
**Problem:** When an addon has degraded apps, it shows "Not Deployed" badge which is confusing — the addon IS deployed, it's just unhealthy.

**Fix:** Review the status logic. "Not Deployed" should only show when there's no ArgoCD app at all. Degraded apps should show "Degraded" with red badge, not "Not Deployed".

**Root cause:** Likely the API returns "missing" health for apps on disconnected clusters (can't reach them = "missing"), which the UI maps to "Not Deployed".

---

## Bug 5: Apps on disconnected clusters show wrong health (core issue)
**Problem:** ArgoCD apps on unreachable clusters have `ComparisonError` and `UnknownError` conditions, but the API returns their health as "Healthy" or "Unknown" without surfacing the actual errors.

**Example:** `hello-world-poultrysense-dev` has:
- `ComparisonError: Failed to load live state: failed to get cluster info... i/o timeout`
- `UnknownError: error synchronizing cache state... i/o timeout`

But the platform shows it as healthy or ignores the conditions entirely.

**What needs to happen:**
1. Backend: fetch app conditions from ArgoCD API (`.status.conditions[]`)
2. If conditions contain `ComparisonError` or `UnknownError`, health should be "Error" not "Healthy"
3. Surface the condition messages in the UI (cluster detail + addon detail)
4. Version matrix should show these as red/amber, not green

**This is the root cause of bugs 3, 4, and 7.** Fix this and they all improve.

---

## Bug 6: Migration branches not deleted after merge
**Problem:** After a migration PR is merged (steps 3, 5, 10), the source branch (`aap/migration/...`) remains in the repo.

**Fix:** Call `gitProvider.DeleteBranch()` after successful merge in `createPRWithLog`. If it fails (permissions), log a warning but don't fail the step.

**Check:** Does `GitProvider` interface already have `DeleteBranch`?

**Effort:** 30 min

---

## Bug 7: Version matrix shows green for broken apps
**Problem:** Same root cause as bug 5 — apps on disconnected clusters show as "Healthy" (green) in the version matrix when they're actually unreachable.

**Fix:** Same fix as bug 5. Once health correctly reflects ArgoCD conditions, the version matrix will show the right colors.
